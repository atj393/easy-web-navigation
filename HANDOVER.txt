OVERLAY BEHAVIOR (packages/focus-overlay/src/index.ts)
  FocusOverlayController API: mount, unmount, isMounted, highlightElement,
  clearHighlight, clearAll, updateForElement, destroy.
  - One extension-owned <div> appended to document.body, open shadow root for
    style isolation, position:fixed, pointer-events:none, z-index max,
    aria-hidden=true.
  - Highlight boxes positioned from getBoundingClientRect, keyed by id
    ("focus" live, "locate" temporary with durationMs auto-clear).
  - Repositions on scroll/resize (rAF-throttled). Disconnected elements auto
    -clear. Null/disconnected inputs are safe (return false).
  - NEVER mutates inspected nodes. unmount/destroy removes container + all
    window listeners + timers.

FOCUS HELPER BEHAVIOR (content script)
  - TOGGLE_FOCUS_HELPER enabled -> overlay.mount() + capture-phase focusin
    listener -> overlay.updateForElement(target); also reflects current
    activeElement. Disabled -> remove listener + overlay.unmount().
  - Never calls focus()/preventDefault(); does not change tab order or ARIA.

ISSUE LOCATE BEHAVIOR (content script)
  - LOCATE_ISSUE { selector } -> findBySelector (document.querySelector, then
    one-level open shadow-root fallback). If found: scrollIntoView (only on
    this user action) + highlightElement id "locate" for ~2000 ms. Returns
    LOCATE_RESULT { found }.

POPUP CHANGES (apps/extension/entrypoints/popup/App.tsx)

  - Focus helper toggle (aria-pressed), best-effort state sync on open via

IMPORTANT DECISIONS
  - Overlay lives in the content script's owned controller; popup only sends
    messages. Focus tracking (focusin) is in the content script; repositioning
    on scroll/resize is in the controller (it tracks elements).
  - Shadow root used for style isolation; box-sizing border-box; focus = solid
    blue, locate = dashed orange.
  - getBoundingClientRect mocked in tests (jsdom has no layout).
  - GET_FOCUS_HELPER_STATE on popup open is best-effort and must NOT inject, to
    honor "prefer user-triggered actions".
  - missing-visible-focus stays deferred: no reliable conservative heuristic.
    Phase 0C provides a focus helper but does not score native focus visibility.

KNOWN LIMITATIONS
  - Overlay is visual-only; not a fix and not a compliance result.
  - Restricted pages (chrome://, etc.) -> friendly failure.
  - Closed shadow DOM / cross-origin iframes not observed; locate resolves in
    main document + open shadow roots only.
  - Locate is best-effort if the page changed since the scan.
  - Firefox build is MV2 (WXT default); Chrome/Edge MV3. No icons yet.

WHAT WAS INTENTIONALLY NOT IMPLEMENTED
  - No assist mode, no tab-path visualization.
  - No auto-fix, no inspected-node mutation, no ARIA injection, no added
    tabindex/keyboard handlers/skip links, no focus stealing, no tab-order
    changes.
  - No auto-scroll except on the explicit user-triggered locate.
  - No AI, no speech, no broad host permissions, no remote, no publish, no PR.

RECOMMENDED MANUAL BROWSER WALKTHROUGH
  1. pnpm dev (loads chrome-mv3 dev build).
  2. Serve demos: pnpm -C apps/demo-sites preview (http://localhost:4321).
  3. Open fake-buttons-page.html, "Scan current page".
  4. Click "Locate on page" on an issue -> a temporary orange highlight appears
     and the element scrolls into view.
  5. Click "Show focus helper"; Tab / Shift+Tab -> blue rectangle tracks focus.
  6. Click "Hide focus helper" -> overlay removed.
  7. Refresh the page -> no stale overlay remains.
  8. Open a chrome:// page -> friendly failure, no crash.
  9. Export report -> keywise-report.md, no compliance claim.
