ROOT CAUSE
  computeTabPath built the keyboard path from collectFocusableElements ->
  sortFocusableElementsForTabOrder. The shared hidden test (isHidden) only
  covered input[type=hidden], the hidden attribute, aria-hidden=true,
  display:none, and visibility:hidden|collapse. A closed application sidebar
  (e.g. ChatGPT) stays in the DOM and focusable while being collapsed via
  zero size, content-visibility:hidden, clipping by an overflow ancestor, or a
  full off-canvas translate. None of those were detected, so hidden sidebar
  controls still received numbered markers.

VISIBILITY-FILTER DECISION
  Added a dedicated, site-independent predicate in @easy-web-navigation/
  dom-scanner: isVisuallyAvailableForKeyboardPath(ctx, el). The scanner's
  focusability semantics (isFocusable/isHidden) were intentionally NOT changed.
  computeTabPath now filters the sorted tab order through this predicate BEFORE
  computing totalDetected and applying the 100/250/500 marker cap, so the
  summary total and the overlay both reflect only visible controls. The filter
  is read-only and never alters the page or its tab order.

  The predicate excludes a control when it (or an ancestor) is hidden /
  aria-hidden / display:none / visibility:hidden|collapse, disabled, inside an
  [inert] subtree, or content-visibility:hidden; or, when real layout is
  available, when it has no rendered client rects, no meaningful size, is fully
  off-canvas HORIZONTALLY, or is fully clipped by a non-document
  overflow:hidden|clip ancestor. Pure geometry helpers (hasMeaningfulSize,
  isFullyOffCanvasHorizontally, isClippedAwayBy) are exported and unit-tested
  with mocked rectangles. Geometry runs only when isLayoutAvailable(ctx) is
  true, so jsdom/headless falls back to DOM/style signals instead of hiding
  everything; the predicate never throws on exotic/detached nodes (defaults to
  keeping the control). It is generic - no class/id/text/product heuristics and
  no ChatGPT-specific checks. The same filter applies to manual activation,
  popup-reopen reapply, This website / All websites automatic checking, and SPA
  route refresh, because they all flow through computeTabPath.

EXPLICIT: NORMAL VERTICALLY OFF-SCREEN CONTROLS REMAIN INCLUDED
  Controls above or below the current viewport on a normal long page are NOT
  excluded. Only HORIZONTAL full off-canvas (the drawer/sidebar pattern) is a
  reason to exclude. Controls inside normal scrollable areas (overflow:auto|
  scroll) are kept; html/body are never treated as clipping ancestors.

EXPLICIT: OPACITY ALONE IS NOT TREATED AS HIDDEN
  opacity:0 is never a reason to exclude a control - some real, keyboard-
  focusable controls intentionally use opacity:0.

KNOWN LIMITATIONS
  - Geometry checks (size/off-canvas/clipping) require a real browser; jsdom and
    other layout-less environments fall back to DOM/style signals only.
  - Clipping detection considers non-scrolling overflow:hidden|clip ancestors;
    unusual transform/scale/filter-based hiding may not be perfectly modeled.
  - A collapsed area that becomes visible again needs a refresh/re-enable (or a
    monitoring refresh) before its controls reappear - markers are a snapshot.
  - "This page only" automatic checking still does not continue across
    navigation (intended browser-permission behavior; explained in the popup).
  - Real product screenshots still pending; manual cross-browser QA still
    required before store submission. Local folder still keywise-web.

WHAT WAS INTENTIONALLY NOT IMPLEMENTED
  - No new permissions; no scanner-rule change; no AI/speech/analytics/remote/
    auto-fix/tracking; no ARIA injection; no page mutation; no tab-order change.
  - No site-specific (ChatGPT or other) selectors or class/id/text heuristics.
  - 100/250/500 marker-limit feature kept unchanged; off-viewport page controls
    still included. No store submission.
